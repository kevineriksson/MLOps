What happened? The v1 model was trained on January only. When evaluated on combined Jan+Feb data, performance degrades. February has different travel patterns (post-holiday, weather changes, different demand). The model has never seen these patterns — this is data drift. MAE increased by 12.8% and R² dropped from 0.79 to 0.72, showing the model struggles to generalize to unseen data distributions.

7. Production Monitoring (Task 4)

Assuming v3 is deployed in production, the following metrics should be monitored:

CategoryMetricDescription
Data Feature Distribution Drift (PSI) Monitor Population Stability Index on input features (PULocationID, pickup_hour, passenger_count). If PSI > 0.2, the incoming data has shifted significantly from training data, and model assumptions may no longer hold.
Model Performance MAE on live predictions Compare predicted fares to actual fares once trips complete. If rolling MAE increases beyond ~20% above the v3 test MAE ($3.77 × 1.2 = $4.53), the model is degrading.
System Prediction latency (p95) Track the 95th percentile response time for model inference. Spikes indicate infrastructure issues, data pipeline failures, or model serving problems that affect user experience.

When to Retrain

Retrain when the rolling MAE on production data consistently exceeds the threshold ($4.53) for one or more weeks, or when PSI on key features exceeds 0.2. This indicates the data distribution has shifted enough that the model needs to learn new patterns.

When to Roll Back

Roll back immediately if a newly deployed model performs worse than the previous version on the same recent data. With DVC this is straightforward:

git checkout v2.0 && dvc checkout

This restores both the code and the exact model/data artifacts from that version.
